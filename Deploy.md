# Deploying to Render — step by step

Goal: turn this folder into a public HTTPS URL you can put in your Sagan submission.

**Estimated time:** 25–35 minutes (most of it is Render's first build).

---

## What you'll have at the end

- A URL like `https://aw-portal.onrender.com` — public, HTTPS, works 24/7.
- Auto-deploys on every `git push` to your main branch.
- A `AW_SECRET_KEY` generated by Render, never committed.
- One caveat: free-tier servers **sleep after 15 minutes of no traffic** and take ~30 seconds to wake up on the first visit. We'll mitigate this in the submission email.

## Step 1 — Create a GitHub repo (5 min)

If you don't have GitHub CLI installed, the web flow is fine:

1. Go to https://github.com/new
2. Repo name: `aw-portal` (or anything you like)
3. **Private** is fine — Render can read private repos.
4. Don't initialize with README/.gitignore/license — we have our own.
5. Click **Create repository**.
6. Copy the SSH or HTTPS URL.

Then in PowerShell:

```powershell
cd C:\Users\Administrator\Documents\karen-test\aw-portal

git init
git add .
git status     # sanity check — make sure aw_portal.db and venv/ are NOT listed
git commit -m "Initial commit: AW Client Report Portal"
git branch -M main
git remote add origin https://github.com/YOUR_USERNAME/aw-portal.git
git push -u origin main
```

If `git status` shows `aw_portal.db` or `venv/`, your `.gitignore` isn't being picked up — check the file is named exactly `.gitignore` (with the leading dot).

---

## Step 2 — Sign up / log in to Render (2 min)

1. Go to https://render.com/
2. Click **Get Started for Free**.
3. Sign up with your **GitHub** account — this is the easiest path because it pre-authorizes Render to read your repos.
4. No credit card needed for the free tier.

---

## Step 3 — Create the service (5 min)

1. Once logged in, click **New +** → **Blueprint** in the top-right of the Render dashboard.
   - "Blueprint" is Render's name for `render.yaml`-driven setup. It auto-fills the form for you.
2. **Connect a repository** → pick `aw-portal`.
3. Render will read `render.yaml` and show you a preview:
   - Name: `aw-portal`
   - Plan: Free
   - Runtime: Docker
   - It already filled in `AW_SECRET_KEY` (generated) and the health-check path.
4. Click **Apply**.

If Render asks to install the Render GitHub App, allow it for the `aw-portal` repo (not all repos — keeps the permission scope tight).

---

## Step 4 — Wait for the first build (10–15 min)

The first build is slow because:

- Render pulls the Playwright Python base image (~2GB).
- pip installs Flask + Playwright + Gunicorn.
- The image is then cached — subsequent deploys take ~2 min.

You'll see live logs in the Render dashboard. Watch for:

```
Successfully built abc123…
Deploy succeeded
```

When the dashboard shows a green **Live** badge, the URL is up. It'll be:

```
https://aw-portal.onrender.com
```

(or with a random suffix if the name was taken).

---

## Step 5 — Smoke-test the live URL (3 min)

1. Open the URL.
2. You should land on the **Login** page.
3. Click **Register**, create the first user (e.g. `admin@firm.local` / something memorable).
4. You'll be redirected to **Clients** (empty list).
5. Click **Seed demo client**.
6. Click **Generate report →** on the demo client.
7. Submit the pre-filled form → preview page → click **Download SACS PDF**. PDF should download.
8. Check **History** — your generated report is there.

If steps 1–8 all work: you're done. Send the URL.

## Troubleshooting

### Build fails with "no such file or directory"

Check `git status` after `git add .` — make sure your Dockerfile, requirements.txt, and the `templates/` and `static/` folders are all staged.

### Live URL shows "Internal Server Error"

Open the Render dashboard → **Logs** tab. Most common causes:

- **Memory limit hit during PDF render.** Free tier is 512MB, Chromium's tight in there. Refresh; it usually completes on retry. If it consistently fails, upgrade to the $7/mo Starter plan (1GB RAM).
- **Database file locked.** Free-tier disk is ephemeral and reset on every deploy — totally fine, but if you happen to redeploy mid-request you'll see this. Just refresh.

### Site sleeps and the recruiter hits a 30-second wait

Two options:

1. **Mention it in the email** (preferred) — we did, in the template above.
2. **Pre-warm before they look** — visit the URL once 60 seconds before you expect them to. Render pings stay warm for ~15 min.

### I want the database to persist across deploys

Free tier doesn't have persistent disks. Options:

- Add a Render Disk (1GB, $0.25/mo) and mount it at `/app/data`, then change `DB_FILENAME` in `db.py` to `"/app/data/aw_portal.db"`.
- Switch to PostgreSQL (Render has a free Postgres tier). Bigger code change — leave it for V2.

For a 2-minute reviewer demo, ephemeral SQLite is fine: they register, click around, leave a fingerprint of activity that disappears next deploy.

## What `render.yaml` configured for you

For reference (you don't have to do anything with this; it just runs):

| Field | Value | Why |
|---|---|---|
| `runtime` | `docker` | Uses our Dockerfile (Playwright base image). |
| `plan` | `free` | $0/mo, 512MB RAM, sleeps after idle. |
| `healthCheckPath` | `/login` | Render pings this; 200 means healthy. |
| `autoDeploy` | `true` | Push to main = auto-deploy. |
| `AW_SECRET_KEY` | auto-generated | Used by Flask sessions + auth. |
| `PYTHONUNBUFFERED` | `1` | Logs appear in real-time, not buffered. |
